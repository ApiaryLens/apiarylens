# ApiaryLens 0.1.0-rc.3

## Release Status

> Superseded — do not deploy. Use [0.1.0-rc.4](0.1.0-rc.4.md). rc.3 remains available only as an immutable audit record.

This was a corrective MVP acceptance candidate, not a stable release. It supersedes
`0.1.0-rc.2`, whose exact published Cloudflare bundle passed provisioning but failed
Scout Bee's final health check because the Worker had been bundled against a stale
compiled contracts package and reported `0.1.0-rc.1`. The manifest, source commit,
deployment variables, and artifact identity all reported `rc.2`, so Scout Bee stopped
safely instead of accepting the inconsistent deployment.

Do not deploy `rc.2`. It remains published only as an immutable audit record. The
complete server lifecycle evidence from `rc.1` remains valid for unchanged migration,
synchronization, backup, restore, and deployment contracts. `rc.3` must pass public
verification, signing, publication, and both profile smoke tests before hands-on
acceptance continues.

## Changes from rc.2

- Rebuilds the shared contracts package before the Worker and PWA.
- Makes release artifact assembly build Worker, Cloudflare PWA, and Windows Scout Bee
  inputs in one ordered command.
- Injects the exact source commit, UTC build time, deployment profile, and artifact
  identity into the Cloudflare PWA build.
- Refuses to package a Worker whose compiled runtime version differs from the release
  manifest.
- Refuses to package a PWA that does not contain the exact manifest version, source
  commit, build time, and artifact identity.
- Removes the Windows command-shell dependency from release and supply-chain child
  processes.

## Upgrade and Recovery

Back up before updating. `rc.3` keeps API `1.0`, sync `1`, database migration `0004`,
deployment plan `1`, export `1`, and local store `1`; no new data migration is
introduced. Verify the content-addressed artifact and repository attestation, stage
the update, preserve pending PWA work, activate, then run health, authenticated
record, media, synchronization, and authorization smoke checks. Use application
rollback only while the schema remains compatible; otherwise restore the verified
pre-update backup.

## Support Window

Release candidates receive best-effort support until replaced by a newer candidate
or the first stable release. Report security defects and data-loss risks privately
according to `SECURITY.md`.
