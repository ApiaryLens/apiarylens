# ApiaryLens 0.1.0-rc.2

## Release Status

This is the audited MVP acceptance candidate, not a stable release. It supersedes
`0.1.0-rc.1` for the remaining physical-device, accessibility, and project-owner
acceptance journeys. The complete server lifecycle evidence from `rc.1` remains
valid for unchanged migration, synchronization, backup, restore, and deployment
contracts; `rc.2` still requires immutable-artifact signing, publication, and
profile smoke checks before those journeys continue.

## Changes from rc.1

- Added Cloudflare parity for organization-scoped direct record lookup.
- Added an explicit Cloudflare `media:read` permission boundary.
- Added D1/R2-backed cross-family negative tests for record reads and writes, sync,
  members, original and thumbnail media, deletion, export, and revoked memberships.
- Expanded the generated OpenAPI document to cover bootstrap status, recovery,
  session rotation, sign-out, members, direct record lookup, and thumbnails.
- Corrected OpenAPI security declarations for public identity operations and
  session-plus-CSRF mutations.
- Added the dated route-by-route authorization and deployment-exposure audit; no
  critical or high finding remains open from that review.

## Included MVP

- Installable offline-capable React PWA with IndexedDB records, drafts, staged
  photos, synchronization, conflicts, and safe update behavior
- Built-in owner, beekeeper, and viewer accounts with protected first-owner claim,
  opaque sessions, invitations, recovery codes, and server-side authorization
- Complete family-to-professional apiary records, history, private media, and owned
  data export
- Portable Node/SQLite/Filesystem Compose and Cloudflare Worker/D1/R2 profiles
- Scout Bee guided install, update, backup, restore, diagnostics, and uninstall
- Public `.org`, `.app`, and `.dev` frontends with synchronized user, operator,
  developer, security, support, and release documentation

## Remaining Acceptance Gates

- Physical iPhone, iPad, and computer offline/relaunch/synchronization journey
- Separate-device invitation acceptance and shared-family verification
- Pending offline inspection and photos preserved exactly once across update
- Manual keyboard, NVDA, VoiceOver, 200% zoom, and forced-colors review
- Final project-owner product, visual, documentation, and stable-release acceptance

## Upgrade and Recovery

Back up before updating. `rc.2` keeps API `1.0`, sync `1`, database migration `0004`,
deployment plan `1`, export `1`, and local store `1`; no new data migration is
introduced. Verify the content-addressed artifact and repository attestation, stage
the update, preserve pending PWA work, activate, then run health, authenticated
record, media, synchronization, and authorization smoke checks. Use application
rollback only while the schema remains compatible; otherwise restore the verified
pre-update backup.

## Support Window

Release candidates receive best-effort support until replaced by a newer candidate
or the first stable release. Report security defects and data-loss risks privately
according to `SECURITY.md`.
